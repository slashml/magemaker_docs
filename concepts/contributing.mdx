## Welcome to the Magemaker Contributing Guide

We're thrilled that you're interested in helping make Magemaker better! This guide explains our preferred workflow, quality standards, and the resources available to contributors.

## Development Setup

<Steps>
  <Step title="Fork & Clone">
  ```bash
  git clone https://github.com/YOUR_USERNAME/magemaker.git
  cd magemaker
  ```
  </Step>

  <Step title="Install Python Dependencies (Editable)">
  ```bash
  pip install -e ".[dev]"
  ```
  </Step>

  <Step title="Preview Docs Locally (Mintlify)">
  ```bash
  npm i -g mintlify   # one-time install
  mintlify dev        # from repo root
  ```
  </Step>

  <Step title="Create a Feature Branch">
  ```bash
  git checkout -b feat/your-feature-name
  ```
  </Step>
</Steps>

## Development Guidelines

### Code Style & Linting

We enforce:

* **Black** – formatting
* **isort** – import order
* **flake8** – linting
* **pytest** – testing

```bash
black .
isort .
flake8
pytest -q   # run tests before every commit
```

### Tests

<Note>
All new functionality **must** include unit and/or integration tests. Use `pytest` and place tests under the closest `test_*.py` file in the appropriate module.
</Note>

### Documentation

1. Update existing markdown / MDX docs to reflect code changes.
2. If you add a new surface area (e.g., a new CLI flag or FastAPI route), create a **new page** under the appropriate section. See the new `/concepts/api-reference.mdx` page for an example.
3. Verify navigation order in `mint.json` if you add a top-level page.

## Pull Request Process

<Steps>
  <Step title="Commit Changes">
  Use [Conventional Commits](https://www.conventionalcommits.org/) for commit messages.
  ```bash
  git add .
  git commit -m "feat(api): add /v1/embeddings endpoint"
  ```
  </Step>
  <Step title="Push & Open PR">
  ```bash
  git push origin feat/api-embeddings
  ```
  Then open a PR against `slashml/magemaker:main`.
  </Step>
  <Step title="CI Checks">
  Our GitHub Actions run linting, tests, and build the docs. Fix any failures.
  </Step>
</Steps>

### Pull Request Checklist

<CardGroup cols={2}>
  <Card title="Clear Description" icon="pen">
    Explain *what* and *why*.
  </Card>
  <Card title="Tests" icon="vial">

  <Card title="Documentation" icon="book">

  <Card title="Small Commits" icon="clock-rotate-left">
    Keep commit history focused.
  </Card>
</CardGroup>

## Commit Message Convention

We follow **Conventional Commits**:

- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation only changes
- `style:` Non-functional style changes
- `refactor:` Code refactoring
- `test:` Adding or updating tests
- `chore:` Build or tooling updates

Example:
```bash
feat(deployment): add support for custom Docker images
```
