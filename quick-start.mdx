## Interactive View

1. Run Magemaker with your desired cloud provider:

```sh
magemaker --cloud [aws|gcp|azure|all]
```

Supported providers:

- `--cloud aws` AWS SageMaker deployment
- `--cloud gcp` Google Cloud Vertex AI deployment
- `--cloud azure` Azure Machine Learning deployment
- `--cloud all` Configure all three providers at the same time

## MageMaker CLI: Querying with YAML Configurations

MageMaker now supports querying deployed models using user-defined YAML configuration files via a new command-line interface option. This enhancement introduces a dedicated argument to streamline and standardize how users interact with model endpoints for inference. Below, we outline the newly added functionality and provide practical usage guidance.

---

### High-Level Overview

**What's New?**

A new command-line argument, `--query`, has been integrated into the MageMaker CLI. This argument allows users to specify a path to a YAML configuration file that defines a query against a deployed model. This approach facilitates reproducible, readable, and version-controllable inference requests, aligning with modern MLOps practices.

---

### Detailed Change Descriptions

#### 1. New `--query` Command-Line Argument

- **Flag:** `--query`
- **Purpose:** Accepts the path to a YAML query configuration file.
- **Type:** `str` (string path to the YAML file)
- **Usage Context:** Used to run inference queries against deployed models.
- **Implementation Details:**
  - Introduced as an argument to the CLI parser.
  - Utilizes a custom action (`action='query'`) to trigger appropriate handler logic.
  - Accepts API key information via `api_key="j2i3udh38hd983d38d3dh383"` (may be used internally for authentication).
- **Help Text:**  
  ```
  path to YAML query configuration file
  ```

#### 2. Placement and Compatibility

- The `--query` argument is now available alongside other CLI options (e.g., `--hf` for Hugging Face model deployments).
- The command can be combined with other flags but is primarily intended for querying inference endpoints.

---

### Usage Examples

#### Querying with a YAML Configuration File

Suppose you have exported or written a query configuration at `.magemaker_config/your-model-query.yaml`, containing input data and invocation parameters for your model.

##### Command Structure

```bash
magemaker --query .magemaker_config/your-model-query.yaml
```

**Explanation:**
- `magemaker`: Invokes the MageMaker CLI.
- `--query`: Specifies that you wish to send an inference request.
- `.magemaker_config/your-model-query.yaml`: Path to your YAML configuration file defining the query details.

##### Sample YAML Query Configuration

A sample YAML file for a text classification model might look like:

```yaml
endpoint: my-classification-endpoint
input:
  text: "Classify this sample sentence."
parameters:
  top_k: 3
```

**Note:** The exact schema may depend on your deployed model and MageMaker's expected format.

---

### Best Practices

- **Version Control:** Store your YAML query files in source control for auditability and reproducibility.
- **Sensitive Data:** Do not include sensitive data (like API keys) in your query YAML files unless you have secured your repository.
- **Validation:** Ensure your YAML configuration matches the input schema expected by your deployed model.

---

### Additional References

- For deploying models, see the usage of `--hf` for Hugging Face models and other deployment options in MageMaker's documentation.
- For more advanced querying and automation, explore MageMaker's YAML schema and API reference.

---

With this feature, MageMaker further streamlines MLOps workflows, making model querying transparent, repeatable, and easy to manage.

### YAML-based Deployment (Recommended)

For reproducible deployments, use YAML configuration:

```sh
magemaker --deploy .magemaker_config/your-model.yaml
```

Example YAML for AWS deployment:

```yaml
deployment: !Deployment
  destination: aws 
  endpoint_name: facebook-opt-test
  instance_count: 1
  instance_type: ml.m5.xlarge
  num_gpus: null
  quantization: null
models:
  - !Model
    id: facebook/opt-125m
    location: null
    predict: null
    source: huggingface
    task: text-generation
    version: null
```

For GCP Vertex AI:

```yaml
deployment: !Deployment
  destination: gcp
  endpoint_name: facebook-opt-test
  accelerator_count: 1
  instance_type: g2-standard-12
  accelerator_type: NVIDIA_L4
  num_gpus: null
  quantization: null

models:
  - !Model
    id: facebook/opt-125m
    location: null
    predict: null
    source: huggingface
    task: null
    version: null
```

For Azure ML:

```yaml
deployment: !Deployment
  destination: azure
  endpoint_name: facebook-opt-test
  instance_count: 1
  instance_type: Standard_DS3_v2
models:
  - !Model
    id: facebook--opt-125m
    location: null
    predict: null
    source: huggingface
    task: text-generation
    version: null
```
<Note>
  The model ids for Azure are different from AWS and GCP. Make sure to use the one provided by Azure in the Azure Model Catalog. 

  To find the relevant model id, follow the following steps

      <Step title="Select Hugging Face in the Collections List">

    	![Azure ML Creation](../Images/hugging-face.png)
    </Step>

  </Steps>
</Note>



### Model Fine-tuning

Fine-tune models using the `train` command:

```sh
magemaker --train .magemaker_config/train-config.yaml
```

Example training configuration:

```yaml
training: !Training
  destination: aws # or gcp, azure
  instance_type: ml.p3.2xlarge # varies by cloud provider
  instance_count: 1
  training_input_path: s3://your-bucket/data.csv
  hyperparameters: !Hyperparameters
    epochs: 3
    per_device_train_batch_size: 32
    learning_rate: 2e-5
```
